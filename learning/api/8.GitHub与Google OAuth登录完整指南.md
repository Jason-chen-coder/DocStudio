# GitHub 与 Google OAuth 登录完整指南

本文档深入讲解 OAuth 2.0 的原理，以及如何在 NestJS + Next.js 技术栈中实现 GitHub 和 Google 第三方登录。

## 1. OAuth 2.0 基础概念

### 什么是 OAuth 2.0？

OAuth 2.0 是一个**授权框架**（不是认证协议），允许第三方应用在用户授权的情况下，访问用户在另一个服务上的资源，**而无需暴露用户的密码**。

> 类比：你去酒店入住，前台给你一张房卡（Token）。你用房卡只能开自己的房间（限定权限），而不是拿到酒店的万能钥匙（密码）。

### 四个核心角色

```
┌─────────────────┐     ┌──────────────────┐
│  Resource Owner  │     │  Resource Server  │
│  (用户本人)       │     │  (GitHub/Google   │
│                  │     │   的用户数据 API)   │
└────────┬────────┘     └────────▲─────────┘
         │                       │
         │ 授权                   │ 携带 Token 请求资源
         ▼                       │
┌─────────────────┐     ┌───────┴──────────┐
│     Client       │     │ Authorization    │
│  (你的应用       │────▶│    Server         │
│   doc_studio)    │     │ (GitHub/Google    │
│                  │◀────│  的授权服务)       │
└─────────────────┘     └──────────────────┘
```

| 角色 | 说明 | 示例 |
|------|------|------|
| **Resource Owner** | 拥有数据的用户 | 使用你 App 的用户 |
| **Client** | 请求访问资源的应用 | doc_studio（你的应用） |
| **Authorization Server** | 负责验证用户身份、颁发 Token | GitHub OAuth / Google OAuth |
| **Resource Server** | 存储用户资源的服务器 | GitHub API / Google API |

### OAuth 2.0 的四种授权模式

| 模式 | 适用场景 | 安全性 |
|------|----------|--------|
| **Authorization Code（授权码模式）** | Web 应用（有后端） | 最高 |
| Implicit（隐式模式） | 纯前端 SPA（已不推荐） | 低 |
| Resource Owner Password | 高度信任的应用 | 中 |
| Client Credentials | 服务间通信（无用户参与） | 中 |

**我们使用的是 Authorization Code（授权码模式）**，这是最安全也是最常用的模式。

## 2. Authorization Code 模式完整流程

### 时序图

```
用户浏览器          你的前端(Next.js)       你的后端(NestJS)      OAuth Provider(GitHub/Google)
    │                    │                      │                         │
    │  1. 点击"GitHub登录" │                      │                         │
    │───────────────────▶│                      │                         │
    │                    │  2. 重定向到后端       │                         │
    │                    │─────────────────────▶│                         │
    │                    │                      │  3. 302 重定向到 OAuth   │
    │◀─────────────────────────────────────────│─────────────────────────▶│
    │                    │                      │                         │
    │  4. 用户在 GitHub/Google 页面登录并授权                               │
    │──────────────────────────────────────────────────────────────────▶│
    │                    │                      │                         │
    │  5. OAuth Provider 带 code 重定向回 callback                        │
    │◀─────────────────────────────────────────────────────────────────│
    │─────────────────────────────────────────▶│                         │
    │                    │                      │  6. 后端用 code 换 token │
    │                    │                      │────────────────────────▶│
    │                    │                      │◀────────────────────────│
    │                    │                      │  7. access_token 返回    │
    │                    │                      │                         │
    │                    │                      │  8. 用 token 获取用户信息 │
    │                    │                      │────────────────────────▶│
    │                    │                      │◀────────────────────────│
    │                    │                      │  9. 用户 profile 返回    │
    │                    │                      │                         │
    │                    │                      │ 10. 创建/更新用户，生成 JWT│
    │                    │  11. 重定向到前端，     │                         │
    │◀─────────────────────────────────────────│  携带 JWT               │
    │                    │                      │                         │
    │  12. 前端存储 JWT   │                      │                         │
    │───────────────────▶│                      │                         │
```

### 分步详解

#### Step 1-3: 发起 OAuth 请求

用户点击"GitHub 登录"按钮，前端跳转到后端的 `/auth/github` 端点。后端构造 OAuth 授权 URL 并重定向：

```
https://github.com/login/oauth/authorize?
  client_id=你的应用ID
  &redirect_uri=http://localhost:3001/auth/github/callback
  &scope=user:email
```

参数说明：
- `client_id`: 在 OAuth Provider 注册应用时获得的 ID
- `redirect_uri`: 授权完成后的回调地址，必须与注册时填写的一致
- `scope`: 请求的权限范围（GitHub 的 `user:email` 表示读取用户邮箱）

#### Step 4: 用户授权

用户在 GitHub/Google 页面上看到"xxx 应用请求访问你的账户"，点击同意。

#### Step 5: 携带授权码回调

OAuth Provider 将用户重定向回你的 callback 地址，URL 中附带一个一次性的 `code`：

```
http://localhost:3001/auth/github/callback?code=abc123xyz
```

> **为什么不直接返回 Token？** 因为这个重定向经过浏览器，URL 中的 Token 会暴露在浏览器历史、日志中。code 是一次性的、短时效的，更安全。

#### Step 6-7: 用 code 换 access_token（后端到后端）

后端在**服务端**向 OAuth Provider 发起 POST 请求，用 `code` + `client_secret` 换取 `access_token`：

```bash
POST https://github.com/login/oauth/access_token
Body:
  client_id=你的应用ID
  client_secret=你的应用密钥（只有后端知道！）
  code=abc123xyz
  redirect_uri=http://localhost:3001/auth/github/callback
```

> **关键安全点**: `client_secret` 绝对不能暴露在前端代码中。这一步必须在后端完成。

#### Step 8-9: 获取用户信息

后端使用 `access_token` 调用 OAuth Provider 的用户 API：

```bash
# GitHub
GET https://api.github.com/user
Authorization: Bearer <access_token>

# Google
GET https://www.googleapis.com/oauth2/v1/userinfo?alt=json
Authorization: Bearer <access_token>
```

返回用户的 profile 信息（邮箱、头像、昵称等）。

#### Step 10-12: 创建用户并签发 JWT

后端拿到用户信息后：
1. 查询数据库中是否已存在该 OAuth 用户（通过 `githubId` / `googleId`）
2. 不存在则创建新用户，存在则更新信息
3. 签发自己应用的 JWT（access_token + refresh_token）
4. 重定向到前端，URL 中携带 JWT

```
http://localhost:3000/auth/oauth-callback?token=xxx&refresh_token=yyy
```

前端的 `oauth-callback` 页面解析 URL 参数，将 JWT 存入 localStorage/cookie。

## 3. GitHub OAuth 与 Google OAuth 对比

### 注册应用

| 项目 | GitHub | Google |
|------|--------|--------|
| 注册地址 | Settings > Developer settings > OAuth Apps | Google Cloud Console > APIs & Services > Credentials |
| 获得的凭证 | Client ID + Client Secret | Client ID + Client Secret |
| 回调地址设置 | Authorization callback URL（单个） | Authorized redirect URIs（可多个） |
| scope 格式 | `user:email` | `email profile` |

### API 端点对比

| 操作 | GitHub | Google |
|------|--------|--------|
| 授权页面 | `https://github.com/login/oauth/authorize` | `https://accounts.google.com/o/oauth2/v2/auth` |
| 换 Token | `POST https://github.com/login/oauth/access_token` | `POST https://oauth2.googleapis.com/token` |
| 获取用户信息 | `GET https://api.github.com/user` | `GET https://www.googleapis.com/oauth2/v1/userinfo` |

### 返回数据结构

```typescript
// GitHub profile
{
  id: 12345678,              // GitHub 用户 ID
  login: "jasonchen",        // 用户名
  email: "jason@example.com",
  name: "Jason Chen",
  avatar_url: "https://avatars.githubusercontent.com/u/12345678"
}

// Google profile
{
  id: "108234567890123456789", // Google 用户 ID
  email: "jason@gmail.com",
  name: "Jason Chen",
  picture: "https://lh3.googleusercontent.com/a/xxxxx"
}
```

## 4. 在 NestJS 中的实现

### 4.1 环境变量配置

```bash
# .env
# GitHub OAuth
GITHUB_CLIENT_ID=your_github_client_id
GITHUB_CLIENT_SECRET=your_github_client_secret
GITHUB_CALLBACK_URL=http://localhost:3001/auth/github/callback

# Google OAuth
GOOGLE_CLIENT_ID=your_google_client_id
GOOGLE_CLIENT_SECRET=your_google_client_secret
GOOGLE_CALLBACK_URL=http://localhost:3001/auth/google/callback

# 前端地址（用于回调重定向）
FRONTEND_URL=http://localhost:3000
```

### 4.2 Passport Strategy 方式（标准方式）

NestJS 通过 Passport.js 提供了统一的 OAuth 抽象。每个 Provider 对应一个 Strategy：

```typescript
// github.strategy.ts
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { Strategy } from 'passport-github2';

@Injectable()
export class GithubStrategy extends PassportStrategy(Strategy, 'github') {
  constructor() {
    super({
      clientID: process.env.GITHUB_CLIENT_ID || '',
      clientSecret: process.env.GITHUB_CLIENT_SECRET || '',
      callbackURL: `${process.env.API_URL}/auth/github/callback`,
      scope: ['user:email'],
    });
  }

  // Passport 在拿到 access_token 和 profile 后自动调用此方法
  // 返回值会挂在 req.user 上
  async validate(accessToken: string, refreshToken: string, profile: any) {
    const { id, emails, displayName, username, photos } = profile;
    return {
      githubId: id,
      email: emails?.[0]?.value,
      name: displayName || username,
      avatarUrl: photos?.[0]?.value,
    };
  }
}
```

```typescript
// google.strategy.ts
import { Injectable } from '@nestjs/common';
import { PassportStrategy } from '@nestjs/passport';
import { Strategy } from 'passport-google-oauth20';

@Injectable()
export class GoogleStrategy extends PassportStrategy(Strategy, 'google') {
  constructor() {
    super({
      clientID: process.env.GOOGLE_CLIENT_ID || '',
      clientSecret: process.env.GOOGLE_CLIENT_SECRET || '',
      callbackURL: `${process.env.API_URL}/auth/google/callback`,
      scope: ['email', 'profile'],
    });
  }

  async validate(accessToken: string, refreshToken: string, profile: any) {
    const { id, emails, displayName, photos } = profile;
    return {
      googleId: id,
      email: emails?.[0]?.value,
      name: displayName,
      avatarUrl: photos?.[0]?.value,
    };
  }
}
```

### 4.3 手动实现方式（适用于 Fastify）

由于 Passport 的重定向机制依赖 Express 的 `res.redirect()`，在 Fastify 下不兼容。因此可以手动实现 OAuth 流程：

```typescript
// auth.controller.ts — Google OAuth 手动实现

// Step 1: 构造授权 URL 并重定向
@Get('google')
async googleLogin(@Res() res: any) {
  const params = new URLSearchParams({
    client_id: process.env.GOOGLE_CLIENT_ID || '',
    redirect_uri: process.env.GOOGLE_CALLBACK_URL || '',
    response_type: 'code',
    scope: 'email profile',
  });
  const url = `https://accounts.google.com/o/oauth2/v2/auth?${params}`;
  return res.code(302).header('Location', url).send();
}

// Step 2: 处理回调
@Get('google/callback')
async googleCallback(@Query('code') code: string, @Res() res: any) {
  const frontendUrl = process.env.FRONTEND_URL || 'http://localhost:3000';

  if (!code) {
    return res.code(302).header('Location', `${frontendUrl}/auth/login?error=oauth_failed`).send();
  }

  // Step 3: 用 code 换 access_token
  const tokenRes = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      code,
      client_id: process.env.GOOGLE_CLIENT_ID || '',
      client_secret: process.env.GOOGLE_CLIENT_SECRET || '',
      redirect_uri: process.env.GOOGLE_CALLBACK_URL || '',
      grant_type: 'authorization_code',
    }),
  });
  const tokenData: any = await tokenRes.json();

  // Step 4: 用 access_token 获取用户信息
  const profileRes = await fetch(
    'https://www.googleapis.com/oauth2/v1/userinfo?alt=json',
    { headers: { Authorization: `Bearer ${tokenData.access_token}` } },
  );
  const profile: any = await profileRes.json();

  // Step 5: 创建/更新用户，签发 JWT
  const result = await this.authService.oauthLogin({
    googleId: profile.id,
    email: profile.email,
    name: profile.name,
    avatarUrl: profile.picture,
  });

  // Step 6: 重定向到前端，携带 JWT
  const url = `${frontendUrl}/auth/oauth-callback?token=${result.access_token}&refresh_token=${result.refresh_token}`;
  return res.code(302).header('Location', url).send();
}
```

### 4.4 数据库模型设计

用户表需要支持多种登录方式共存：

```prisma
model User {
  id            String   @id @default(uuid())
  email         String   @unique
  password      String?           // OAuth 用户可能没有密码
  name          String?
  avatarUrl     String?

  // OAuth 字段（各 Provider 的唯一标识）
  googleId      String?  @unique
  githubId      String?  @unique

  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
}
```

**关键设计决策：**
- `password` 设为可选（`?`），因为纯 OAuth 用户没有密码
- 每个 OAuth Provider 用独立字段（`googleId`, `githubId`）标识
- `email` 作为统一标识——同一邮箱的 OAuth 和密码注册可以合并账户

### 4.5 AuthService 中的 oauthLogin 逻辑

```typescript
async oauthLogin(profile: {
  googleId?: string;
  githubId?: string;
  email: string;
  name?: string;
  avatarUrl?: string;
}) {
  // 1. 先通过 OAuth ID 查找
  let user = await this.prisma.user.findFirst({
    where: profile.googleId
      ? { googleId: profile.googleId }
      : { githubId: profile.githubId },
  });

  if (!user) {
    // 2. 再通过 email 查找（合并已有账户）
    user = await this.prisma.user.findUnique({
      where: { email: profile.email },
    });

    if (user) {
      // 绑定 OAuth 到已有账户
      user = await this.prisma.user.update({
        where: { id: user.id },
        data: profile.googleId
          ? { googleId: profile.googleId }
          : { githubId: profile.githubId },
      });
    } else {
      // 3. 创建新用户
      user = await this.prisma.user.create({
        data: {
          email: profile.email,
          name: profile.name,
          avatarUrl: profile.avatarUrl,
          googleId: profile.googleId,
          githubId: profile.githubId,
        },
      });
    }
  }

  // 4. 签发 JWT
  const tokens = this.generateTokens(user);
  return tokens;
}
```

```tsx
// app/auth/oauth-callback/page.tsx
'use client';

import { useEffect } from 'react';
import { useSearchParams, useRouter } from 'next/navigation';

export default function OAuthCallbackPage() {
  const searchParams = useSearchParams();
  const router = useRouter();

  useEffect(() => {
    const token = searchParams.get('token');
    const refreshToken = searchParams.get('refresh_token');

    if (token && refreshToken) {
      // 存储 Token
      localStorage.setItem('access_token', token);
      localStorage.setItem('refresh_token', refreshToken);
      // 跳转到首页
      router.replace('/');
    } else {
      // 登录失败
      router.replace('/auth/login?error=oauth_failed');
    }
  }, [searchParams, router]);

  return <div>登录中...</div>;
}
```

## 6. 安全注意事项

### 6.1 必须遵守的安全原则

| 原则 | 说明 |
|------|------|
| **client_secret 只在后端** | 永远不要将 secret 暴露到前端代码或 Git 仓库中 |
| **验证 state 参数** | 防止 CSRF 攻击。每次请求生成随机 state，回调时验证一致性 |
| **使用 HTTPS** | 生产环境必须全程 HTTPS，防止 Token 在传输中被截获 |
| **Token 短时效** | access_token 设置较短过期时间（15min~1h），配合 refresh_token 续期 |
| **验证 redirect_uri** | 后端只允许预注册的回调地址，防止开放重定向攻击 |

### 6.2 CSRF 防护：state 参数

```typescript
// 发起请求时生成 state
@Get('github')
async githubLogin(@Res() res: any) {
  const state = crypto.randomUUID(); // 随机值
  // 存储到 session 或 cookie 中
  res.setCookie('oauth_state', state, { httpOnly: true, maxAge: 600 });

  const params = new URLSearchParams({
    client_id: process.env.GITHUB_CLIENT_ID || '',
    redirect_uri: '...',
    scope: 'user:email',
    state,  // 附加 state
  });
  // ...重定向
}

// 回调时验证 state
@Get('github/callback')
async githubCallback(@Query('state') state: string, @Req() req: any) {
  const savedState = req.cookies['oauth_state'];
  if (state !== savedState) {
    throw new UnauthorizedException('Invalid state parameter');
  }
  // ...继续处理
}
```

### 6.3 Token 安全存储

| 存储方式 | 优点 | 缺点 |
|----------|------|------|
| localStorage | 简单易用 | 容易受 XSS 攻击 |
| httpOnly Cookie | 前端 JS 无法读取，防 XSS | 需处理 CSRF |
| 内存（变量） | 最安全 | 刷新页面丢失 |

**推荐方案**: httpOnly Cookie 存储 refresh_token，内存中存 access_token，刷新页面时用 refresh_token 换新。

## 7. 常见问题与排错

### Q1: 回调地址不匹配

**错误**: `redirect_uri_mismatch`

**原因**: OAuth Provider 注册的回调地址和代码中的不一致。

**排查**:
- GitHub: Settings > Developer settings > OAuth Apps > 检查 callback URL
- Google: Cloud Console > Credentials > 检查 Authorized redirect URIs
- 注意末尾斜杠、http/https、端口号必须完全一致

### Q2: scope 权限不足

GitHub 和 Google 的 scope 格式不同：
- GitHub: `user:email`（冒号分隔）
- Google: `email profile`（空格分隔）

### Q3: 本地开发 HTTPS 问题

Google OAuth 在本地开发时允许 `http://localhost`，但不允许 `http://127.0.0.1`。确保回调地址使用 `localhost`。

### Q4: OAuth 用户后续想设置密码

提供"设置密码"功能，让纯 OAuth 用户补充密码，后续可以用邮箱+密码登录：

```typescript
async setPassword(userId: string, newPassword: string) {
  const hashed = await bcrypt.hash(newPassword, 10);
  return this.prisma.user.update({
    where: { id: userId },
    data: { password: hashed },
  });
}
```

```
├──────────────────────────────────────────────────────────────────┤

│                              [后端 /auth/github/callback]         │
│                                          │                       │
│                              ┌───────────┴──────────┐            │
│                              │   后端 ──▶ GitHub     │            │
│                              │   code + secret       │            │
│                              │      换 token         │            │

│                              └───────────┬──────────┘            │

│                                          │                       │

│                              存储 JWT，跳转首页                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**核心要点**:
1. 用户的密码永远不经过你的应用
2. `code` 是一次性的，`client_secret` 只在后端使用
3. OAuth 拿到的是 Provider 的 `access_token`，最终签发的是你自己应用的 JWT
4. 同一邮箱的多种登录方式可以合并到同一用户账户
